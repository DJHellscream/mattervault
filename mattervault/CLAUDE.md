# Mattervault - Private Document Intelligence System

The 'agent' - You - claude code - is a Law Firm Product Architect and understands the workflow of actual Law Firm employees.

## 1. Project Overview
Mattervault is a **private, air-gapped Document Intelligence System** built for legal and estate planning. It ingests PDFs, preserves their structure (tables, headers) using high-fidelity parsing, and enables semantic search and chat ("RAG") without ever sending data to the cloud.

## 2. Architecture & Networking

### The "Hybrid" Stack
We use **Docker** for orchestration but break out heavy AI compute to the **Native Host (Windows)** to leverage GPU/Metal acceleration and avoid Docker resource bottlenecks.

| Service | Type | Hostname / URL | Port | Notes |
| :--- | :--- | :--- | :--- | :--- |
| **Paperless-ngx** | Docker | `webserver` | `8000` | The Vault. Stores "Original" & "Archived" PDF/A. |
| **n8n** | Docker | `n8n` | `5678` | The Brain. Orchestrates ingestion & Chat UI. |
| **Qdrant** | Docker | `qdrant` | `6333` | The Memory. Vector DB (Collection: `mattervault_documents`). |
| **Postgres** | Docker | `db-paperless`, `db-n8n` | `5432` | Isolated DBs for Vault and Brain. |
| **Redis** | Docker | `redis` | `6379` | Cache & Message Broker. |
| **Ollama** | **Native** | `host.docker.internal` | `11434` | **External.** Runs LLMs & Embeddings on Windows. |
| **Docling** | **Native** | `host.docker.internal` | `5001` | **External.** `docling-serve` running on Windows. |

### Network Topology
- **Docker Network:** `matternet` (Bridge)
- **Host Gateway:** Containers access Native apps (Ollama/Docling) via `host.docker.internal`.
- **Intake:** Host folder `./intake` is mapped to container path `/usr/src/paperless/consume/intake`.

## 3. Data Flow (The Pipeline)
1.  **Ingest:** User drops PDF into `./intake` (Windows).
2.  **Archive:** Paperless detects file → OCRs (clean-final) → Stores PDF/A-2.
3.  **Trigger:** n8n Webhook receives "Document Added" event from Paperless.
4.  **Parse:** n8n sends PDF to **Docling** (`http://host.docker.internal:5001/convert`).
    * *Output:* Structured Markdown (preserving legal tables/clauses).
5.  **Embed:** n8n sends chunks to **Ollama** (`nomic-embed-text`).
6.  **Store:** Vectors stored in **Qdrant** (Payload includes: `text`, `file_id`, `page_num`).
7.  **Retrieval:** User chats in n8n UI → Hybrid Search in Qdrant → Answer generated by Ollama (`llama3`).

## 6. RAG Strategy: The "Legal-Grade" Logic
*Since we are running "headless" (n8n only), the following logic must be implemented manually in the workflows.*

### A. Parsing Strategy (Docling)
**Goal:** Preserve legal structure (clauses, definitions, tables).
* **Method:** Do NOT use plain text extraction. Use **Docling** to convert PDF to **Markdown**.
* **Why:** Markdown preserves headers (`## Article 1`), lists, and tables. This allows the LLM to understand that "Section 2.1" belongs to "Article 2".

### B. Chunking Strategy: "Parent-Child" (Hierarchical)
**Goal:** Retrieve the *full* clause, not just the sentence that matched the search.
* **Concept:**
    * **Parent:** The meaningful unit of text (e.g., a full Contract Clause, a Will Article, or a Table). Size: ~1000-2000 tokens.
    * **Child:** The search unit. Small, overlapping chunks derived from the Parent. Size: ~256-512 tokens.
* **Implementation in n8n:**
    1.  **Ingest:** Receive Markdown from Docling.
    2.  **Split:** Use a Code Node (or Split Text Node) to split by **Markdown Headers** (`#`, `##`).
    3.  **Vectorize:** Create embeddings for the *Child* chunks.
    4.  **Storage (Qdrant):**
        * **Vector:** The Child embedding.
        * **Payload (Metadata):** Store the **Full Parent Text** in a field called `context_text`.
        * **Result:** When we search, we match the Child vector but retrieve the Parent `context_text` for the LLM.

### C. Retrieval Strategy: "Hybrid Search"
**Goal:** Catch both specific terms ("Form 1040") and concepts ("Tax obligations").
* **Dense Vector Search:** Uses `nomic-embed-text` (via Ollama). Finds semantic meaning.
* **Keyword/Sparse Search:** (Implemented via Qdrant Payload filtering or BM25 if enabled).
    * *Action:* Always filter retrieval by `file_type` or `date` if the user query implies it.
    * *Refinement:* If Qdrant sparse vectors are unavailable in this MVP, simulate hybrid by boosting results that contain exact keyword matches from the user's query.

### D. Ranking Strategy: "Re-Ranking"
**Goal:** Fix "Lost in the Middle" phenomenon and remove irrelevant matches.
* **The Workflow:**
    1.  **Broad Search:** Retrieve Top 25 results from Qdrant (High Recall).
    2.  **Rerank:** Pass these 25 results + the User Question to a **Cross-Encoder** (or a specialized prompt in Ollama `llama3`).
    3.  **Filter:** Select the Top 5 highest-scored chunks.
    4.  **Generate:** Send only these Top 5 to the LLM for the final answer.
* *Note:* This dramatically reduces hallucinations compared to feeding raw search results directly to the chat.

### E. System Prompt (The "Persona")
* **Role:** "You are Mattervault, an expert legal analyst and estate planning assistant."
* **Constraint:** "Answer ONLY based on the provided Context. If the answer is not in the context, state that you do not know. Do not hallucinate outside legal facts."
* **Citation:** "Always cite the Source Document and Page Number for every claim (e.g., 'Source: Will_JohnDoe.pdf, Page 4')."

## 7. Multi-Tenancy: Family Data Isolation

Estate planning firms handle multiple families (e.g., Morrison, Johnson). Data must NEVER commingle between families.

### Strategy: Payload Filtering (Single Collection)
We use a single Qdrant collection (`mattervault_documents`) with a mandatory `family_id` field for data isolation.

**Qdrant Payload Schema:**
```json
{
  "family_id": "morrison",           // REQUIRED - tenant identifier
  "text": "...",                      // Child chunk text
  "context_text": "...",             // Parent text for retrieval
  "document_id": "paperless_doc_123", // Link back to Paperless
  "document_title": "Morrison Trust 2024",
  "page_num": 4,
  "chunk_index": 12
}
```

**Query-time isolation (MANDATORY):**
```python
# Every search MUST include family_id filter
results = qdrant.search(
    collection_name="mattervault_documents",
    query_vector=embedding,
    filter=Filter(
        must=[FieldCondition(key="family_id", match=MatchValue(value="morrison"))]
    ),
    limit=25
)
```

### Family ID Flow Through the Pipeline

1. **Intake:** Create subfolders per family: `./intake/morrison/`, `./intake/johnson/`
2. **Paperless:** `PAPERLESS_CONSUMER_SUBDIRS_AS_TAGS=true` auto-tags documents by subfolder
3. **Webhook:** Paperless sends tag info to n8n
4. **n8n Workflow:** Extract family tag → Store as `family_id` in Qdrant
5. **Chat:** User selects family at session start → All queries filtered by `family_id`

### Self-Healing Workflow (Tag Correction)
If a document is dropped in the wrong folder, the paralegal can fix the tag in Paperless:
1. **Correction:** Paralegal updates the family tag in Paperless UI
2. **Trigger:** Paperless fires `document.updated` webhook to n8n
3. **Re-index:** n8n deletes old vectors (by `document_id`) and re-indexes with correct `family_id`

This ensures eventual consistency without requiring perfect human behavior at intake.

### Hybrid Search Architecture (V2)

```
User Question
     │
     ▼
┌────────────────┐
│ Embed (dense)  │ ─► nomic-embed-text → 768-dim vector
│ + Tokenize     │ ─► hashCode() → BM25 sparse vector
└────────┬───────┘
         │
         ▼
┌─────────────────────────────────────┐
│        Qdrant Hybrid Search         │
│  ┌───────────┐   ┌───────────┐      │
│  │  Dense    │   │   BM25    │      │
│  │ (semantic)│ + │ (keyword) │      │
│  └─────┬─────┘   └─────┬─────┘      │
│        └───────┬───────┘            │
│                ▼                    │
│         RRF Fusion → Top 15         │
└─────────────────┬───────────────────┘
                  │
                  ▼
┌─────────────────────────────────────┐
│    Batch Reranking (qwen3:4b)       │
│    Score each doc vs query          │
│    Select Top 5 by relevance        │
└─────────────────┬───────────────────┘
                  │
                  ▼
┌─────────────────────────────────────┐
│    LLM Generation (llama3.1:8b)     │
│    Context: Top 5 parent chunks     │
│    Output: Answer with citations    │
└─────────────────────────────────────┘
```

**Why Hybrid Search for Legal Docs:**
- Dense (semantic): Catches conceptual matches ("estate distribution" ↔ "inheritance")
- BM25 (keyword): Catches exact terms ("Section 2.1(a)", "Form 1040")
- RRF Fusion: Combines both for best coverage

## Common Tasks

### Updating n8n Workflows (IMPORTANT - READ THIS)

**The Problem:** Using `n8n import:workflow` without the original workflow ID creates DUPLICATES instead of updating the existing workflow.

**The Solution:** When importing a workflow JSON, if the JSON contains an `id` field that matches an existing workflow, it will OVERWRITE that workflow. If the ID doesn't exist (or is missing), it creates a NEW workflow.

**Correct Procedure to Update an Existing Workflow:**

1. **Export the current workflow to get its ID:**
   ```bash
   # List all workflows to find the ID
   docker exec matterlogic n8n list:workflow

   # Export the specific workflow (note the ID!)
   docker exec matterlogic n8n export:workflow --id=<WORKFLOW_ID> --output=/tmp/workflow.json
   docker cp matterlogic:/tmp/workflow.json ./n8n-workflows/
   ```

2. **Edit the local JSON file with your changes, KEEPING the same `id` field:**
   ```json
   {
     "id": "abc123-original-id-KEEP-THIS",  // <-- DO NOT CHANGE THIS
     "name": "Document Ingestion Pipeline V2",
     "nodes": [ ... ]
   }
   ```

3. **Import the updated workflow (it will overwrite because ID matches):**
   ```bash
   docker cp ./n8n-workflows/my-workflow.json matterlogic:/tmp/
   docker exec matterlogic n8n import:workflow --input=/tmp/my-workflow.json
   ```

4. **Restart n8n to pick up changes:**
   ```bash
   docker restart matterlogic
   ```

**NEVER do this:**
- Delete the `id` field from the JSON (creates duplicate)
- Change the `id` field (creates duplicate with new ID)
- Use `n8n import:workflow` without checking the ID first

**If you accidentally created duplicates:**
1. Go to n8n UI (localhost:5678)
2. Identify the correct workflow (check "Updated" timestamp)
3. Delete the duplicates manually
4. Export the correct one to update local JSON with correct ID

## Known Issues

## Known "Gotchas"
- Networking: If n8n cannot talk to Ollama, ensure Ollama is running with OLLAMA_HOST=0.0.0.0 on Windows.
- Intake: Paperless only watches /usr/src/paperless/consume. We map ./intake to /usr/src/paperless/consume/intake.
- PDF Parsing: Docling is CPU/GPU intensive. Large legal docs (>100 pages) may timeout; increase n8n HTTP timeout settings to 300s.
